Manifesto Técnico · v0.1

LLM-First Architecture uma arquitetura pensada para ser mantida por IA

Um método de organização de projetos web backend que prioriza a legibilidade por agentes de IA — sem sacrificar boas práticas de engenharia de software. Não é um framework. É um protocolo de organização e documentação.

01
O Problema que Este Método Resolve

A memória que se perde entre sessões

O problema da memória perdida

Quando uma LLM é usada para manter um projeto ao longo do tempo, ela enfrenta um problema estrutural grave: ela não tem memória entre sessões. Isso cria um ciclo destrutivo que degrada o projeto progressivamente.

O desenvolvedor que usa IA como assistente passa cada vez mais tempo corrigindo inconsistências e cada vez menos tempo criando valor.

O problema da ambiguidade

Mesmo dentro de uma única sessão, uma LLM enfrenta ambiguidade quando a estrutura do projeto não é explícita. Quando a estrutura não diz claramente onde cada coisa deve viver, a IA adivinha.

E adivinhações acumuladas ao longo de sessões destroem a coesão do projeto.

ciclo-destrutivo.txt 
Sessão 1  →  LLM entende o projeto do zero, toma boas decisões
Sessão 2  →  LLM reaprende parcialmente, decisões levemente inconsistentes
Sessão 3  →  LLM contradiz decisões anteriores sem saber
Sessão N  →  Código vira uma colcha de retalhos incoerente

"O futuro do desenvolvimento assistido por IA não está nos modelos mais poderosos. Está na qualidade do contexto que fornecemos a eles."

O que é

LLM-First Architecture

LLM-First Architecture resolve os dois problemas com uma abordagem dupla: estrutura de pastas previsível onde cada camada tem responsabilidade única e explícita, e documentação viva por camada onde cada pasta carrega seu próprio contexto técnico em um arquivo CONTEXT.md.

03
As regras que tornam o método funcional

Princípios Fundamentais

Todo projeto que segue LLM-First Architecture deve respeitar os seguintes princípios. Eles não são sugestões — são as regras que tornam o método funcional.

1

Responsabilidade Única por Arquivo

Cada arquivo tem uma responsabilidade clara e nomeada. Um arquivo de rotas não contém lógica de negócio. Essa separação é o que permite à LLM saber onde agir sem ambiguidade.

2

Dependência Unidirecional

As camadas se comunicam em uma única direção: de fora para dentro. Nenhuma camada interna conhece as externas. O projeto se torna um grafo acíclico de dependências — previsível por definição.

3

Core Imutável

O core/ é o núcleo que sustenta tudo. Modificá-lo é uma decisão de arquitetura. Uma LLM nunca deve modificar o core para implementar uma feature.

4

Contexto Explícito

Nenhuma decisão de arquitetura deve existir apenas na cabeça do desenvolvedor. Toda decisão relevante é documentada no CONTEXT.md da camada afetada. O que não está documentado não existe para a LLM.

5

Documentação como Contrato

O CONTEXT.md define o que pode, o que deve e o que não pode acontecer naquela camada. Uma LLM que viola o contrato está cometendo um erro arquitetural — mesmo que o código funcione.

6

Atualização Obrigatória

Toda modificação feita no código deve ser refletida no CONTEXT.md da pasta afetada antes do encerramento da sessão. Documentação desatualizada é pior que ausência de documentação.

05
O elemento central do método

Sistema CONTEXT.md

O CONTEXT.md é o elemento mais inovador do LLM-First Architecture. Ele transforma cada pasta em uma unidade autossuficiente de contexto — a memória persistente daquela camada da aplicação.

📋

O que existe aqui?

Lista dos arquivos com descrição de uma linha — o que cada um faz e como.

🔧

Como foi implementado?

Decisões técnicas e suas justificativas — libs usadas, padrões adotados.

⛔

O que NÃO fazer aqui?

Restrições explícitas — o elemento mais valioso para LLMs. Elimina erros arquiteturais.

📅

O que mudou recentemente?

Changelog por camada — a LLM lê e sabe o estado atual sem precisar do git log.

Por que restrições explícitas são o elemento mais valioso

Documentar o que não deve ser feito é mais valioso do que documentar o que deve. Uma LLM que sabe que core/security.py não deve acessar o banco de dados nunca vai cometer esse erro — mesmo que tecnicamente seja possível.

Restrições explícitas eliminam a categoria de erros mais difícil de detectar: o código que funciona mas está no lugar errado.

06
Templates completos

Anatomia do CONTEXT.md

Existem dois tipos de CONTEXT.md: o raiz (mapa do projeto inteiro) e os de subpasta (contrato técnico de cada camada).

6.1 — CONTEXT.md Raiz

O primeiro arquivo que uma LLM deve ler ao iniciar qualquer sessão. Responde: "Qual é este projeto, como está organizado e quais são as regras globais?"

CONTEXT.md — Raiz 
# [Nome do Projeto] — CONTEXT.md Raiz

## Identidade do Projeto
- **Nome:** [nome]
- **Tipo:** API REST / Web App / Híbrido
- **Stack principal:** [linguagem + framework + banco]
- **Autenticação:** [JWT / OAuth2 / Session]
- **Versão atual:** [v0.x]

## Estrutura de Camadas

| Pasta       | Responsabilidade                          |
|-------------|-------------------------------------------|
| `core/`     | Configuração, segurança, banco            |
| `api/`      | Rotas HTTP — recebe, valida e delega      |
| `services/` | Lógica de negócio — não conhece HTTP     |
| `models/`   | Entidades ORM — espelho do banco          |
| `schemas/`  | Contratos Pydantic de entrada e saída     |

## Regra de Dependência Global

api/ → services/ → models/ + schemas/
            ↓
          core/

Nenhuma camada importa de uma camada acima dela.

## Restrições Globais

⛔ Nunca colocar lógica de negócio em api/
⛔ Nunca importar de api/ ou services/ dentro de core/
⛔ Nunca acessar o banco diretamente em api/
⛔ Nunca hardcodar segredos no código

## Histórico de Decisões

| Data       | Decisão                  | Motivo              |
|------------|--------------------------|---------------------|
| AAAA-MM-DD | Estrutura inicial criada | Início do projeto   |

Guia para a LLM

Esta seção é escrita para a LLM — é o guia de comportamento esperado de qualquer agente que opere em um projeto LLM-First.

Antes de qualquer ação

→ SEMPRE ler o CONTEXT.md da raiz primeiro
→ SEMPRE ler o CONTEXT.md da pasta afetada
→ NUNCA assumir o que uma camada faz sem ler

Ao criar um novo arquivo

→ Identificar em qual camada pertence
→ Verificar se já existe algo similar
→ Atualizar CONTEXT.md após criar

Árvore de decisão para ambiguidade

onde-colocar.decision-tree 
Se não souber onde um código deve viver:

1. Esse código conhece HTTP?
   └─ Sim → pertence a api/

2. Esse código é uma regra de negócio?
   └─ Sim → pertence a services/

3. Esse código é uma função pura sem estado?
   └─ Sim → pertence a utils/

4. Esse código define a estrutura do banco?
   └─ Sim → pertence a models/

5. Esse código valida entrada ou define saída?
   └─ Sim → pertence a schemas/

Ainda há dúvida? → Reportar ao desenvolvedor antes de agir.

Contexto no ecossistema

Comparações

MVC Tradicional

Separação em Model, View, Controller. Controller tende a acumular lógica. Documentação é externa ao código. Decisões são implícitas no histórico.

LLM-First Architecture

Camadas adicionais (services, schemas) isolam a lógica de negócio. Documentação é embutida em cada camada. Decisões são explícitas e auditáveis.

Clean Architecture

Alta complexidade com muitas camadas e interfaces. Curva de aprendizado íngreme. Não prescreve documentação. Foco em independência de frameworks.

LLM-First Architecture

Complexidade moderada sem over-engineering. Curva suave. Documentação é central ao método. Foco em legibilidade por agentes de IA.

Quando usar LLM-First

🤖

Projetos com IA recorrente

Mantidos com assistência de IA de forma contínua ao longo do tempo.

👥

Times pequenos

Onde a IA é um colaborador real e o contexto precisa ser compartilhado.

📅

Projetos de longo prazo

Onde consistência arquitetural ao longo do tempo é crítica.

🔄

Alta rotatividade de contexto

Múltiplos desenvolvedores, múltiplas ferramentas de IA, múltiplas sessões.

Os erros mais comuns ao ignorar o método

Anti-Padrões

Anti-padrões são os erros mais comuns cometidos quando o método não é seguido — ou é seguido apenas parcialmente. Reconhecê-los é tão importante quanto conhecer as boas práticas.

Documentação desatualizada é pior que ausência de documentação.

⛔

Anti-Padrão 1 — CONTEXT.md desatualizado

O arquivo existe, mas não foi atualizado após a última modificação. A LLM lê, confia e age com base em informação errada — produzindo código inconsistente com o estado real do projeto.

Solução: atualizar o CONTEXT.md é parte da tarefa, não um passo opcional após ela. O encerramento de qualquer sessão só está completo quando os arquivos CONTEXT.md das pastas afetadas estão atualizados.

⛔

Anti-Padrão 2 — Lógica de negócio em api/

O arquivo de rotas começa a acumular condicionais, cálculos e decisões de domínio. A LLM que recebe uma tarefa nova não sabe se deve colocar a lógica no service ou na rota — e escolhe a rota porque já tem lógica lá.

O problema se amplifica a cada sessão até o arquivo de rotas se tornar impossível de manter.

Solução: api/ deve ter apenas: receber, validar com schema, chamar o service e retornar resposta. Qualquer if de regra de negócio pertence a services/.

⛔

Anti-Padrão 3 — core/ modificado para implementar features

Um desenvolvedor pede à LLM para "adicionar validação de e-mail" e ela coloca a lógica em core/security.py porque parece relacionado à segurança. O core agora conhece regras de negócio — e qualquer outra camada que importar o core carrega esse acoplamento junto.

Solução: core/ é infraestrutura pura: configuração, conexão com banco e autenticação primitiva. Validações de domínio ficam em services/ ou schemas/.

⛔

Anti-Padrão 4 — utils/ como gaveta de miscellaneous

Com o tempo, utils.py vira um arquivo de 800 linhas com funções sem relação entre si. A LLM não sabe o que está lá sem ler tudo — e passa a duplicar funções que já existem ou ignora o arquivo completamente.

Solução: utils/ deve ser uma pasta com arquivos nomeados por função: email.py, formatters.py, validators.py. Se um helper tem lógica de domínio, ele pertence ao service daquele domínio — não a utils.

⛔

Anti-Padrão 5 — Iniciar sessão sem ler o contexto

O desenvolvedor vai direto ao ponto sem usar o Prompt Padrão de Sessão. A LLM executa — mas sem saber que aquele model tem uma restrição documentada no CONTEXT.md, ou que o schema relacionado precisa ser atualizado junto.

O código funciona, mas o projeto perde coerência silenciosamente. Sessão após sessão, o acúmulo de decisões tomadas sem contexto destrói a arquitetura.

Solução: sempre usar o Prompt Padrão de Sessão. A leitura do contexto antes da ação é o que separa uma sessão produtiva de uma que gera débito técnico invisível.

Da teoria à prática

Exemplo End-to-End de uma Sessão Real

A teoria faz sentido — mas como é uma sessão LLM-First na prática? O exemplo a seguir mostra o ciclo completo para uma tarefa concreta: implementar recuperação de senha por e-mail.

Passo 1 — Dev abre a sessão com o Prompt Padrão

O desenvolvedor usa o prompt de abertura padrão informando a tarefa:

Prompt de Abertura — Exemplo
Você está operando em um projeto que segue o LLM-First Architecture. Antes de qualquer ação, leia o CONTEXT.md da raiz e o CONTEXT.md de cada pasta afetada. Tarefa: implementar recuperação de senha via e-mail com link de token temporário.

Passo 2 — LLM lê o CONTEXT.md da raiz

A LLM descobre as informações essenciais do projeto:

  • Stack: FastAPI + PostgreSQL + JWT
  • Autenticação centralizada em core/security.py
  • Envio de e-mail já existe em utils/email.py
  • Regras de negócio sempre em services/
  • Restrição global: nunca acessar o banco diretamente em api/

Passo 3 — LLM identifica as camadas afetadas e lê seus CONTEXT.md

A LLM mapeia todas as pastas que a tarefa vai tocar e lê o CONTEXT.md de cada uma:

Pasta Motivo O que o CONTEXT.md revela
models/ Novo model de token Padrão de nomenclatura e como declarar entidades
schemas/ Novo contrato de request Padrão de naming dos schemas
services/ Lógica de geração e validação Restrição: não retornar HTTPException aqui
api/ Nova rota Restrição: sem lógica de negócio
utils/ Envio de e-mail email.py já existe com função enviar_email()

Passo 4 — LLM reporta o plano antes de agir

Antes de escrever qualquer código, a LLM apresenta o plano:

plano-implementacao-reset-senha.txt 
Plano de implementação:

1. models/password_reset.py     → criar model PasswordResetToken
2. schemas/auth_schema.py       → adicionar PasswordResetRequest e PasswordResetConfirm
3. services/auth_service.py     → adicionar solicitar_reset() e confirmar_reset()
4. api/auth.py                  → adicionar POST /auth/reset e POST /auth/reset/confirm
5. utils/email.py               → adicionar enviar_email_reset()

Observação: core/security.py já tem criar_token() — será reutilizado,
não será criada função duplicada. Isso está alinhado com a restrição
documentada no CONTEXT.md do core.

Passo 5 — LLM implementa camada por camada, na ordem correta

A implementação segue a ordem do fluxo de dependência — de dentro para fora:

  • models/password_reset.py — base de dados definida primeiro.
  • schemas/auth_schema.py — contrato de entrada/saída definido.
  • services/auth_service.py — lógica implementada sobre models e schemas já definidos.
  • api/auth.py — rota criada sobre o service já pronto.

Cada camada só é escrita após a anterior estar definida. Isso evita retrabalho e garante que as dependências estejam corretas.

Passo 6 — LLM atualiza os CONTEXT.md de cada pasta tocada

Antes de encerrar a sessão, a LLM atualiza cada CONTEXT.md afetado:

  • models/CONTEXT.md — adiciona password_reset.py na tabela de arquivos.
  • schemas/CONTEXT.md — registra os dois novos schemas.
  • services/CONTEXT.md — documenta solicitar_reset() e confirmar_reset().
  • api/CONTEXT.md — registra as duas novas rotas.
  • utils/CONTEXT.md — registra enviar_email_reset() em email.py.

Passo 7 — Próxima sessão começa do mesmo ponto

Qualquer LLM — ou o mesmo modelo em uma nova conversa — lê os CONTEXT.md e sabe exatamente:

  • O que existe em cada camada.
  • Como foi implementado e por quê.
  • O que não deve ser alterado.

Resultado: zero perda de contexto entre sessões.

Limites claros do método

O que este método não resolve

Nenhum método é solução universal. LLM-First Architecture resolve problemas específicos de contexto e previsibilidade — mas existe um conjunto claro de problemas que estão fora do seu escopo. Conhecer esses limites é essencial para aplicar o método com expectativas corretas.

🧠

Não substitui um desenvolvedor experiente

O método organiza o contexto para a LLM agir com mais precisão — mas decisões de arquitetura de alto nível, escolha de tecnologias e trade-offs de produto ainda exigem julgamento humano.

A estrutura de pastas pode ser perfeita e o CONTEXT.md impecável — mas se a decisão de negócio por trás da feature estiver errada, o código estará errado de forma organizada.

A IA executa melhor. Não julga por você.

🧪

Não substitui testes automatizados

CONTEXT.md documenta intenção e restrições. Ele não garante que o código implementado funciona corretamente.

Um service pode estar no lugar certo, respeitar todas as restrições da camada e ainda assim ter um bug lógico que só um teste unitário vai revelar.

Testes automatizados continuam sendo a única forma de verificar comportamento real do código.

🤖

Não elimina alucinações do modelo

O método reduz significativamente o espaço para erros de contexto ao tornar as responsabilidades explícitas — mas não elimina a possibilidade de a LLM gerar código incorreto, usar uma API de forma errada ou introduzir um bug sutil.

Revisão humana do output continua sendo necessária, especialmente em lógica crítica, financeira ou de segurança.

📐

Não resolve má arquitetura prévia

Aplicar o método em um projeto já desorganizado — com lógica espalhada, dependências circulares e arquivos sem responsabilidade clara — exige refatoração antes de qualquer ganho.

LLM-First Architecture é mais eficaz quando adotado desde o início do projeto.

Migrar um projeto legado caótico para o método é um projeto por si só, e deve ser tratado como tal.

⚖️

Adiciona overhead de manutenção

Manter os CONTEXT.md atualizados tem custo real. Cada sessão que modifica código exige que o desenvolvedor — ou a LLM — dedique tempo para atualizar a documentação das camadas afetadas.

Em projetos muito pequenos, de prototipagem rápida ou de vida curta, esse overhead pode não se justificar. O método cresce em valor proporcional ao tamanho, complexidade e longevidade do projeto.

🔒

Não garante segurança do código

O método organiza onde o código de segurança vive (core/security.py) e documenta suas restrições — mas não faz auditoria de segurança.

Vulnerabilidades em autenticação, injeção de dados, exposição de endpoints e gestão de segredos ainda dependem de revisão especializada e ferramentas dedicadas de análise de segurança.

"Um método honesto sobre seus limites é um método em que você pode confiar nos seus pontos fortes."

A história por trás do método

De onde veio o LLM-First Architecture

Todo método nasce de uma dor real. Esta seção conta a origem do LLM-First Architecture — não como narrativa de marketing, mas como registro honesto do caminho que levou a uma solução que ninguém havia documentado antes.

13

Primeiro contato — curiosidade pura

Aos 13 anos, um computador novo e a internet ainda engatinhando no Brasil. Era a época do Orkut, do MSN, dos jogos online como WYD, MU e Gunbound. A curiosidade natural de qualquer jovem da época levou a uma pergunta que mudaria tudo: como funcionam os cheats e trainers desses jogos? Um vídeo que demorou dois dias para baixar — e que apresentou o Visual Studio 5.0 — foi o primeiro contato com o mundo da programação. Não por obrigação. Por fascínio.

~20

Uma década de aprendizado não linear

Sem formação técnica formal, os anos seguintes foram de exploração autodidata. Servidores privados de jogos rodando em máquina própria — primeiro Gunbound, depois WYD — forçaram o contato precoce com C++, MySQL, Apache, PHP e HTML. Não era estudo estruturado. Era necessidade prática: para o servidor funcionar, o banco de dados precisava funcionar. Foi aí que nasceu um amor que permanece até hoje — a persistência de dados, a elegância de um schema bem modelado, a lógica que vive no banco.

31

A decisão — ADS, São Paulo, 2023

Com 31 anos, a decisão de formalizar o que a curiosidade havia ensinado de forma fragmentada. Início do curso de Análise e Desenvolvimento de Sistemas em março de 2023 — concluído em dezembro de 2025. Três anos de formação acadêmica que confirmaram uma percepção já existente: a lógica sempre foi o ponto forte. Escrever código de backend do zero, o ponto fraco. Arquitetura, regras de negócio, modelagem de dados, infraestrutura — naturais. Implementação linha a linha no backend — lenta, custosa, frequentemente errada. Essa tensão foi o ponto de partida para tudo que veio depois.

2023

Fevereiro de 2023 — o momento que mudou tudo

Três meses após o lançamento do ChatGPT no Brasil, o primeiro contato com o modelo. A reação não foi de curiosidade casual — foi de reconhecimento imediato. Ali estava a resposta para a tensão que existia há anos: a lacuna entre pensar a solução com clareza e conseguir escrever o código que a implementava.

A percepção foi clara desde o início: isso não é uma ferramenta de busca melhorada. É uma nova camada de programação. A questão não era se as LLMs iam mudar o desenvolvimento de software — era entender como essa camada funcionava por dentro para usá-la com precisão, não com sorte.

now

3 anos de estudo — o caminho até o método

Os anos seguintes foram dedicados a entender LLMs com profundidade técnica: limites de janela de contexto de diferentes modelos, NLP aplicado a engenharia de prompt, tentativa de construção de modelo próprio, mais de 200 projetos experimentais com diferentes arquiteturas e abordagens de colaboração com IA. O LLM-First Architecture não foi planejado em uma tarde. Foi destilado de centenas de falhas documentadas — projetos que a IA degradou progressivamente, inconsistências que se acumularam, sessões que perderam o fio condutor. O método é a resposta estrutural a um problema que só quem passou por ele entende completamente.

O paradoxo que gerou o método

Era possível projetar a arquitetura de um sistema complexo, modelar o banco de dados com precisão, definir regras de negócio detalhadas e pensar em infraestrutura com Docker e ambientes separados — mas transformar tudo isso em código backend funcional consumia tempo desproporcional e frequentemente resultava em implementações frágeis.

A lógica estava clara. O código não saía. Essa é uma tensão que mais desenvolvedores experimentam do que admitem — e que as LLMs deveriam resolver, mas não resolviam de forma consistente em projetos com arquitetura real.

A conclusão que gerou a arquitetura

O problema não era a LLM. Era a ausência de estrutura que permitisse à LLM manter coerência ao longo do tempo. Um modelo pode ser brilhante em uma sessão e destrutivo na próxima — não porque ficou menos inteligente, mas porque perdeu o mapa do território.

A solução era criar esse mapa. Não como documentação para humanos lerem depois — mas como contexto estruturado que a IA lê antes de agir. Essa mudança de perspectiva é o núcleo de tudo.

O que este caminho ensinou sobre o futuro do desenvolvimento

A programação evoluiu em camadas de abstração. Cartões perfurados, Assembly, C, linguagens de alto nível, frameworks declarativos — cada salto foi um aumento na distância entre a intenção humana e a instrução da máquina. O prompt estruturado é o próximo salto. Não é uma linguagem no sentido formal — mas é um protocolo de instrução com sintaxe própria, regras implícitas de estrutura e consequências diretas na qualidade do que é produzido.

Quem aprende a operar nessa camada com precisão — entendendo como o modelo processa contexto, onde ele falha, quais restrições precisa receber explicitamente — não está usando IA como ferramenta. Está programando em um nível de abstração mais alto do que qualquer linguagem existente.

O LLM-First Architecture é uma tentativa de formalizar esse nível. De transformar em protocolo o que hoje ainda depende de intuição individual. A visão de longo prazo é mais ambiciosa: um modelo treinado especificamente para operar dentro desta arquitetura como padrão nativo — onde a estrutura de pastas, os CONTEXT.md e o protocolo de sessão não são instruções externas, mas conhecimento interno do modelo. Esse seria o marco de uma nova geração de LLMs voltadas ao desenvolvimento.

A inspiração por trás do método

Este método nasceu de uma observação prática e recorrente: LLMs falham de forma sistemática ao tentar desenvolver e escalar projetos de software com arquiteturas complexas.

Mesmo engenheiros de prompt experientes encontram uma barreira difícil de transpor — à medida que um projeto cresce em número de arquivos e camadas de responsabilidade, a LLM perde a capacidade de manter coerência entre as partes. Ela começa a alucinar contexto: assume responsabilidades erradas para arquivos, cria dependências invertidas, repete lógica em camadas distintas ou quebra funcionalidades existentes ao tentar implementar algo novo.

O problema não é a inteligência do modelo — é a ausência de uma arquitetura projetada para ser lida e mantida por ele. LLM-First Architecture surgiu para preencher exatamente essa lacuna: transformar a estrutura do projeto em contexto explícito, previsível e auditável, para que a IA possa agir com precisão independentemente da complexidade acumulada.
